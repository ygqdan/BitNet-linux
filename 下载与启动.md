### 6.4 提速版（减少 KV 缓存/上下文）

你前面测试时如果发现“输出很慢”，多半是因为 `llama-cli` 默认/模型元信息导致 KV cache 很大（例如 `n_ctx` 过高）。

可以把上下文和批处理尺寸调小来提速（仍然是中文测试题）：

```bash

cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-7B-Instruct/qwen2.5-7b-instruct-q4_k_m.gguf" \
  -p "你能干的最复杂的任务是什么？" \
  -n 90 \
  -c 2048 \

  -b 512 -ub 128 \

  --temp 0.2 --top-k 30 --top-p 0.9 \
```

#### 6.6.3 启动推理（推荐：Flash Attention + q8 KV，明显减轻 KV 带宽）

说明：纯 CPU 跑 14B 仍然偏慢；`-fa` 打开 Flash Attention，`--cache-type-k/v q8_0` 把 KV 从 f16 压到约一半（同样 `-c` 下更少内存与带宽）。线程数可按机器改：`-t $(nproc) -tb $(nproc)`。

**内存偏紧（例如约 16GB RAM、已在用 Swap）时：请优先用下面 6.6.4 的稳妥参数**，避免 `-b 512` + `-c 2048` 在长 prompt 时瞬时吃满内存导致剧烈抖动。

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "你是一个中文技术助手。请用简洁中文回答：Linux 的三个核心特点是什么？" \
  -n 120 \
  -c 2048 \
  -t "$(nproc)" -tb "$(nproc)" \
  -b 512 -ub 128 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 128 --min-p 0.05 \
  --no-warmup --no-perf
```

#### 6.6.4 稳妥少抖动（约 16GB 内存 / 已占用 Swap 时推荐）

思路：14B Q4 权重约 8.4GiB，再加大上下文和 `batch` 容易把剩余内存和磁盘 Swap 顶满，表现为一卡一卡。**收紧 `-c` 降 KV 与注意力开销；收紧 `-b` / `-ub` 降长 prompt 预填充时的峰值内存**。仍保留 `-fa` 与 `q8_0` KV。

可选（需 root，整系统行为）：推理前临时降低换出倾向，减轻「边推边被换到 Swap」的抖动，结束后可改回：

```bash
sudo sysctl vm.swappiness=10
# 用完后：sudo sysctl vm.swappiness=60
```

稳妥启动示例（与你当前机型 16 线程、`free` 显示约 16Gi 内存的场景对齐）：

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "你是一个中文技术助手。请用简洁中文回答：Linux 的三个核心特点是什么？" \
  -n 120 \
  -c 1024 \
  -t "$(nproc)" -tb "$(nproc)" \
  -b 256 -ub 64 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 128 --min-p 0.05 \
  --no-warmup --no-perf
```

若仍偶发卡顿：先把 `-c` 再降到 `768`；或关其它占内存应用后再跑。一般**不建议**在 16GB 上对 14B 再加 `--mlock`（容易与其它进程抢物理页，反增压力）。

#### 6.6.5 进一步提速（短对话 / 内存更宽裕时）

- 在**内存够用**（例如关闭其它大程序、`free` 里可用内存明显高于模型体积）时，可把 `-c` 提到 `2048`、把 `-b` / `-ub` 提到 `512` / `128`，长 prompt 的预填充会更快。
- 若只追求 decode 更快、不需要长上下文，继续用较短 `-c`（如 `768`～`1024`）往往更划算。
- **内存很宽裕**且希望减少权重被换出导致的抖动时，才可试 `--mlock`；16GB 跑 14B 时通常不优先考虑。
- 若有 NVIDIA GPU 且本机构建了 CUDA，`llama-cli` 可加 `-ngl 99`（或更高）把层卸到显存，速度通常远高于纯 CPU。

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "用一句话介绍 Python。" \
  -n 80 \
  -c 1024 \
  -t "$(nproc)" -tb "$(nproc)" \
  -b 512 -ub 128 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 128 --min-p 0.05 \
  --no-warmup --no-perf
```

#### 6.6.6 交互加速（更快出首 token）

如果你觉得“等很久才开始输出”，通常是 **prompt 预填充（prefill）** 在慢。可以把 `-c/-b/-ub/-n` 同步收紧，让本轮需要预填充/写入 KV 的 token 数更少。

建议优先用这一套（仍保留 `-fa` 和 `q8_0` KV）：

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "你是一个中文技术助手。用简洁中文回答：Linux 的三个核心特点是什么？" \
  -n 80 \
  -c 768 \
  -t "$(nproc)" -tb "$(nproc)" \
  -b 128 -ub 32 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 64 --min-p 0.05 \
  --no-warmup --no-perf
```

如果你对输出长度敏感（不想模型“说太长才停”），把 `-n` 再降到 `60`；如果你需要稍长一点，再把 `-n` 回升到 `100`。

#### 6.6.7 长对话避免“越聊越慢”（`--keep` 控制上下文膨胀）

当你把“历史对话”也拼进每次的 prompt 时，随着历史增多，`-c` 对应的 KV 会持续变大，最终会变慢或抖动。你可以用 `--keep` 限制初始 prompt 进入上下文的 token 数量，从而避免越聊越慢。

适用场景：你每轮都在 prompt 里追加大量历史消息（而不是只给当前问题）。如果每次只给一句问题，通常不需要。

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "（把你的长对话拼进这里）请用简洁中文回答最后一个问题。" \
  -n 100 \
  -c 1024 \
  --keep 512 \
  -t "$(nproc)" -tb "$(nproc)" \
  -b 256 -ub 64 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 128 --min-p 0.05 \
  --no-warmup --no-perf
```

#### 6.6.8 线程数微调（让 CPU 更“顺滑”）

对 14B 的纯 CPU 推理，线程用满（`-t $(nproc)`）不一定最快；有时会因为内存带宽/调度竞争变慢或忽快忽慢。建议你用同一组其它参数，只把线程从 16 降到 12 或 8 试一下。

把下面这一行里的 `-t` / `-tb` 分别改成 `12/12` 或 `8/8`：

```bash
-t 12 -tb 12
```

如果你需要一个“稳妥默认”，可以直接用：

```bash

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

  -n 80 \
  -c 768 \
  -t 12 -tb 12 \
  -b 128 -ub 32 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 64 --min-p 0.05 \
  --no-warmup --no-perf
```

#### 6.6.9 增加回复长度上限（字数更多）

你看到“回复字数有限”，通常是因为命令里的 `-n` 设得偏小。`-n` 的含义是：**最多生成多少个 token**（不是固定字数，但中文大致上 token 会与字数同数量级）。

调大 `-n` 的同时要注意：**`-c` 必须留出足够的上下文**，否则可能在还没生成完就因为上下文满而停止。

在你当前的 16GB + 14B 场景，建议这样试：

1. 仍用“稳妥上下文”时：`-c 1024` + `-n 200`
2. 想更省内存/更快首 token：`-c 768` + `-n 150`

示例（更长一点的回复）：

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-14B-Instruct/qwen2.5-14b-instruct-q4_k_m.gguf" \
  -p "你是一个中文技术助手。请用有条理的要点回答：Linux 的三个核心特点是什么？每点给一个例子。" \
  -n 200 \
  -c 1024 \
  -t 12 -tb 12 \
  -b 256 -ub 64 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 128 --min-p 0.05 \
  --no-warmup --no-perf
```

如果你是多轮对话，把历史拼得很长也会“抢上下文空间”；这时优先用 `--keep` 限制进入上下文的历史 token 数量（见上面的 6.6.7）。

```bash
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 64 --min-p 0.05 \
  --no-warmup --no-perf
```

**更快试机**（只生成少量 token，验证能跑通即可）：把 `-n` 改为 `20`。

### 7.3.1 提高输出速度（纯 CPU、16GB）

32B 在 CPU 上主要受 **算力 + 内存带宽** 限制；不加大硬件时，优先从下面几项榨体感速度：

1. **缩短 `-c`**：KV 与每层 attention 随上下文变重；在能跑稳的前提下，把 `-c` 从 `512` 降到 **`384` 或 `320`**，**decode 通常会更快**（只适合短对话/短 prompt；多轮历史会吃满上下文）。
2. **缩短 prompt**：首字慢多半是 **prefill**；提示越短，开始输出越快。
3. **扫一遍线程数**：同一组其它参数，把 `-t/-tb` 在 **6、8、12** 各试一轮短生成，取**最快且不抖**的一组（16 核**不必**用满）。
4. **`-n` 小则整体快**：不需要长回复时降低 `-n`；长文只能更久，这是物理限制。
5. **少 Swap**：关浏览器等大内存程序；必要时临时 `vm.swappiness=10`（与 6.6.4 同类思路）。
6. **若有 NVIDIA + CUDA 构建**：`llama-cli` 加 **`-ngl 99`**（或更高）把层卸到显存，通常远快于纯 CPU。

**偏提速的一键示例**（在 7.3 基础上收紧 `-c`，可按上面对比 `-t`）：

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-32B-Instruct/qwen2.5-32b-instruct-q3_k_m.gguf" \
  -p "用两三句话说明：Linux 在服务器上常见的优势是什么？" \
  -n 60 \
  -c 384 \
  -t 12 -tb 12 \
  -b 64 -ub 16 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 48 --min-p 0.05 \
  --no-warmup --no-perf
```

若仍卡顿或内存顶满：把 **`-c` 再降到 `320`**，并把 **`-t/-tb` 改回 `8`**；prefill 仍慢时可试 **`-b 128 -ub 32`**（略吃内存，长一点的 prompt 有时会快一点）。

### 7.3.2 极限提速（本机纯 CPU 上尽量快；接受能力与取舍）

**结论先说清：** 在 **16GB + 纯 CPU** 上跑 **32B**，“最快”仍然有硬上限；要 **数量级**变快，基本只能 **上 GPU**（本机构建含 CUDA 时用 **`-ngl 99` 或更高**，见文末命令块）。

在只能 CPU 的前提下，把下面几条同时做到，通常最接近本机极限：

1. **`-c` 尽量小**：单轮、极短提问可用 **`256`**（再低容易 prompt+输出撞上上下文）。仍慢再试 **`320`**，略宽松但 decode 会重一点。
2. **prompt 尽量短**：prefill token 少，**首字时间**最短。
3. **`-n` 只留需要的量**：问答一句就够时 **`40`～`60`**。
4. **`-b/-ub` 尽量小**：降低预填充峰值内存，减轻 Swap 抖动；**`-b 32 -ub 8`**（仅适合**很短**的 prompt）。
5. **`-t/-tb` 试一圈**：在 **`6 / 8 / 12`** 里选全程最快、CPU 不 100% 打架 Swap 的一组，没有 universal 最优。
6. **系统侧**：关大内存程序；必要时 **`vm.swappiness=10`**；保证磁盘 Swap 别狂转。

**本机极限倾向的一套（短问题 + 短回答）：**

```bash
cd "/home/srbank/code/BitNet－linux"

export LD_LIBRARY_PATH="$PWD/build/3rdparty/llama.cpp/src:$PWD/build/3rdparty/llama.cpp/ggml/src:$LD_LIBRARY_PATH"

./build/bin/llama-cli \
  -m "models/Qwen2.5-32B-Instruct/qwen2.5-32b-instruct-q3_k_m.gguf" \
  -p "一句话：Linux 适合做服务器吗？" \
  -n 45 \
  -c 256 \
  -t 12 -tb 12 \
  -b 32 -ub 8 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 32 --min-p 0.05 \
  --no-warmup --no-perf
```

若出现加载或生成异常、或 prompt 略长，先把 **`-c` 提到 `320`**，并把 **`-b/-ub` 提到 `64/16`**。

**真正“最快”路线（有 NVIDIA 且 `llama-cli` 支持 CUDA 时）**，在 **与上相同的轻量 `-c/-n` 思路**下增加 GPU 卸载（层数按你显存可改为 `50` 等先试）：

```bash
./build/bin/llama-cli \
  -m "models/Qwen2.5-32B-Instruct/qwen2.5-32b-instruct-q3_k_m.gguf" \
  -p "一句话：Linux 适合做服务器吗？" \
  -n 45 \
  -c 512 \
  -ngl 99 \
  -fa \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.2 --top-k 30 --top-p 0.9 \
  --repeat-penalty 1.1 --repeat-last-n 32 --min-p 0.05 \
  --no-warmup --no-perf
```

（若命令报错不支持 `-ngl`，说明当前构建是纯 CPU，只能退回上一段 CPU 极限参数。）

### 7.4 若速度/内存仍不可接受

- 优先回退到 `Qwen2.5-14B-Instruct`（14B 在 16GB 上实用性更高）。
- 或保持 32B 仅做短回复：`-n 40~80`、`-c 384~512`。
- 合并成功且确认 `qwen2.5-32b-instruct-q3_k_m.gguf` 可用后，可删除 5 个 `*-0000x-of-00005.gguf` 分片以省磁盘（**删前请自行再确认路径**）。
